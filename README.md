# B2C Retail CRM

Окремий проєкт для блоку роздрібної торгівлі.

Цей проєкт не є частиною `marketplace-crm` і не змінює блоки маркетплейсу або B2B.

## Структура

- `index.html` - окрема сторінка B2C роздрібного магазину.
- `styles.css` - стилі тільки для retail-блоку.
- `app.js` - клієнтська логіка роздрібного магазину.
- `server.mjs` - Node.js сервер для зовнішнього доступу, API і спільного server-backed стану.
- `server.ps1` - PowerShell TCP fallback server для старих Windows, де Node.js не запускається.
- `start-server.ps1` - Windows-скрипт запуску серверної версії.
- `deploy-remote.ps1` - локальний deployment на Windows-сервер через PowerShell remoting без Git на сервері.
- `retail-crm.config.example.json` - приклад конфігурації серверного запуску.
- `logs/` - окремі робочі логи B2C.
- `B2C_HANDOFF_2026-06-06.md` - підсумковий файл для продовження роботи в новому чаті.
- `NEW_CHAT_START_MESSAGE.md` - готовий текст, з якого треба почати новий чат.

## Поточний прототип

MVP-прототип містить:

- у блоці `Продаж` поле `Покупець` підтягує довідник `Клієнти і лояльність` і дозволяє відбір тільки всередині поля введення за ім'ям або телефоном, без окремого вікна списку;
- панель KPI магазину;
- єдиний блок `Продаж`, який об'єднує продаж, пошуковий вибір покупця з довідника, обов'язкове підтвердження методу оплати перед проведенням, заборону оплати `Банк` без клієнта з довідника, пошуковий вибір і додавання товару за назвою/SKU/штрихкодом/QR, автоматичну знижку лояльності, рядковий список продажів, друкований перегляд продажу та відкриття/закриття касової зміни;
- повне та часткове повернення по чеку з модальним уточненням, як повернути гроші покупцю: POS або готівка;
- клікабельні KPI-картки й підсумкові вікна `Продажів сьогодні`, `Виторг сьогодні`, `Журнал повернень` і `Звіт дня` з розшифровкою документів, переходом до продажу/повернення та редагуванням документа за дозволом ролі;
- списки продажів, повернень і KPI-розшифровок у вигляді клікабельних стрічок з пошуком за датою та покупцем, сортуванням за стовпцями і стрілкою згортання; ці можливості винесені в окремі дозволи ролей;
- праве випадаюче підменю `Довідники` з товарами тільки з SQL-імпорту, блоком `Клієнти і лояльність` зі списком клієнтів і створенням нового клієнта з рівнем лояльності та залишками магазину;
- інвентаризацію з додаванням товарів з довідника `Товари`, окремою дією додавання всіх товарів із залишками, додаванням поодиноких товарів після заповнення списку, видаленням товару зі списку, роздрібними цінами, пошуком, друком інвентаризаційного листа, фактичними залишками, різницею, barcode/QR скануванням, пересортом плюс/мінус у грошах і проведенням, яке оновлює залишки складу;
- надходження в магазин тільки з SQL-імпорту, без ручного оприбуткування;
- блок працівників з ролями Директор, Адміністратор, Продавець, Касир, картками працівників, зміною ролі працівника, редагуванням дозволів ролей галочками, перемикачем активного працівника і приховуванням блоків/дій без дозволу;
- server-backed режим для першої реальної програми: зовнішній доступ через Node.js сервер, спільний стан для кількох користувачів, серверне збереження даних у `data/`, API `/api/state`, `/api/settings`, `/api/bootstrap`;
- блок `Налаштування` з параметрами сервера, public host, API base URL, сховища, multi-user режиму й автооновлення;
- денний звіт;
- внутрішній журнал операцій.

Відкрити локально можна через `index.html`.

## PostgreSQL one_c_mirror

- B2C використовує SQL-шар PostgreSQL `one_c_mirror`, а не прямі запити до 1C, якщо дані вже є в SQL.
- Прототип стартує без локально створених товарів, клієнтів, залишків, серійних номерів, чеків і касових змін. Довідникові та складські дані з'являються тільки після SQL-імпорту.
- Товари прив'язані до `crm_products`, `crm_product_price_summary`, `crm_product_characteristics`, `crm_product_kinds`, `crm_product_series`, `crm_product_groups`, `crm_product_folders` і показують `product_code`, `product_group_path`, `product_full_path`, `product_group_code_path`, `product_group_level`, ціни, валюти, характеристики, вид, серію та групу.
- Залишки беруться з `crm_stock_balances` і `crm_warehouses`: окремо показуються `Склад №1` (`warehouse_code = 2`), `Склад Гуртовий` і загальний залишок по всіх складах.
- Серійні номери прив'язані до `crm_serial_stock_current`, `crm_serial_stock_by_serial`, `crm_serial_stock_summary` і показують товар, склад, серійний номер, кількість, знак залишку, `product_code` та `warehouse_code`.
- Клієнти/контрагенти прив'язані до `crm_counterparties`, `crm_counterparty_contracts`, `crm_counterparty_settlements`, `crm_counterparty_balance_summary`.
- Для категорій із папок B2C використовує views `one_c_mirror.crm_product_folder_attributes` і `one_c_mirror.crm_products_enriched`, які додані SQL-міграцією `005_one_c_product_folder_attributes.sql`.
- Окремий блок `Обмін даними` показує імпортоване, експортоване, перебіг процесу, журнал обміну і налаштування автоматизації. Доступ до перегляду, імпорту, експорту й автоматизації керується матрицею ролей.
- Імпортні дії зібрані тільки в блоці `Обмін даними`; у довідниках `Товари`, `Клієнти і лояльність` та `Залишки` блоки запуску імпорту не показуються.
- У довіднику `Клієнти і лояльність` дозволене створення нового B2C-клієнта за правом ролі `Створити клієнта`; такі клієнти позначаються як локальні B2C і очікують експорту в `one_c_mirror.b2c_counterparties_export_queue`.

## LAN переглядалка

Для першої реальної багатокористувацької версії запускай серверний режим. На сервері `192.168.0.5`:

```powershell
cd D:\Codex\CRM\retail-crm
powershell -ExecutionPolicy Bypass -File .\start-server.ps1 -HostName 0.0.0.0 -Port 8790 -PublicHost 192.168.0.5 -DataDir data
```

Після запуску відкрий:

```text
http://192.168.0.5:8790/index.html
```

Дані зберігаються на машині, де запущений сервер:

```text
data/retail-crm-state.json
data/retail-crm-settings.json
```

`data/` і `retail-crm.config.json` не комітяться. Для власних параметрів скопіюй `retail-crm.config.example.json` у `retail-crm.config.json`.

`start-server.ps1` спочатку пробує Node.js (`server.mjs`). Якщо Node.js відсутній або не підтримує стару Windows на сервері, скрипт автоматично запускає `server.ps1` як PowerShell TCP server з тими самими API `/api/health`, `/api/bootstrap`, `/api/state`, `/api/settings`.

Якщо на сервері немає Git, розгортай із локального комп'ютера через PowerShell remoting:

```powershell
cd D:\Codex\CRM\retail-crm
$cred = Get-Credential -UserName '192.168.0.5\zahar'
powershell -ExecutionPolicy Bypass -File .\deploy-remote.ps1 -ComputerName 192.168.0.5 -Credential $cred -RemotePath D:\Codex\CRM\retail-crm -Port 8790 -PublicHost 192.168.0.5
```

Якщо на сервері немає Node.js, повтори deployment з bundled Node:

```powershell
powershell -ExecutionPolicy Bypass -File .\deploy-remote.ps1 -ComputerName 192.168.0.5 -Credential $cred -RemotePath D:\Codex\CRM\retail-crm -Port 8790 -PublicHost 192.168.0.5 -IncludeBundledNode
```

Статичну LAN-переглядалку для прототипного режиму теж можна запускати, але вона не дає спільного серверного збереження:

```powershell
cd D:\Codex\CRM\retail-crm
powershell -ExecutionPolicy Bypass -File .\serve-lan.ps1 -Port 8790
```

У виводі скрипт покаже локальну адресу та LAN-адресу. На іншому пристрої відкрий:

```text
http://LAN-IP:8790/index.html
```

Якщо Windows Defender Firewall покаже запит, дозволь доступ для приватної мережі.
